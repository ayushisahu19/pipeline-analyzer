# Design Document

## Overview
The Multi-Branch Pipeline Analyzer is designed to collect and analyze CI/CD pipeline metrics generated by Jenkins. The system automatically gathers pipeline execution data and stores it in a database for further analysis.

The architecture follows a simple DevOps workflow that integrates GitHub, Jenkins, and a backend analytics service.

## System Architecture
The system workflow is as follows:
Developer pushes code → GitHub repository → Jenkins Multibranch Pipeline → Build and Test stages → Metrics sent to API → MongoDB database → Analytics endpoints

Each component plays a specific role in the system.

## Components
### GitHub
GitHub hosts the project repository and manages source code changes. Whenever a developer pushes code to a branch, Jenkins automatically detects the change.

### Jenkins
Jenkins acts as the CI/CD automation tool. It executes the pipeline stages including build, testing,& metrics reporting. Jenkins Multibranch Pipeline automatically creates pipelines for each branch.

### Node.js API
The backend API receives pipeline execution data from Jenkins and stores it in the MongoDB database. It also provides analytics endpoints to retrieve performance insights.

### MongoDB
MongoDB stores information about each pipeline run including the branch name, build time, build status, and vulnerability count.

### Docker
Docker is used to containerize the backend application and database services. This ensures consistent execution across different environments.

## Data Flow
1. A developer pushes code to the GitHub repository.
2. Jenkins detects the new commit and triggers the pipeline.
3. Jenkins executes the build and test stages.
4. After completion, Jenkins sends pipeline metrics to the Node.js API.
5. The API stores this information in MongoDB.
6. Users can access analytics through API endpoints.
